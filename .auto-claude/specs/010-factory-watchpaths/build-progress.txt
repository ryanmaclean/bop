### Pattern Analysis (lines 152-197)

**WatchPaths trigger mechanism:**
- `WatchPaths` array (lines 172-175): specifies directories to monitor
- Fires job on ANY filesystem change (create, modify, delete, rename)
- No polling - launchd uses FSEvents for efficient monitoring

**Job configuration:**
- `ProgramArguments` (lines 177-184): command to execute when triggered
- Each argument is a separate array element (not shell-parsed)
- Uses absolute path to binary for reliability

**Logging:**
- StandardOutPath/StandardErrorPath (lines 186-190): /tmp/*.log files
- Critical for debugging event-driven jobs

**Implicit defaults (keys NOT present):**
- No `KeepAlive` → defaults to false (run-on-demand only)
- No `ThrottleInterval` → no debouncing (fires on every change)
- No `RunAtLoad` → only runs when watched path changes

**Key insight for factory implementation:**
The factory should follow the same pattern but MAY want to add:
- ThrottleInterval to debounce rapid card creation events
- Different logging paths
- Specific command: `bop factory --cards-root <path> --once`

This pattern provides event-driven execution without polling overhead.

- ❌ Had: `--poll-ms 500` and `--reap-ms 1000`
- ✓ Needed: `--once` flag for one-shot execution

**Root cause:** The WatchPaths pattern requires the dispatcher to:
1. Fire on filesystem event (card dropped in pending/)
2. Run once (process available cards)
3. Exit
4. Launchd re-arms for next event

Polling mode (`--poll-ms`) defeats the purpose of event-driven WatchPaths.

extra_args = r#"    <string>--vcs-engine</string>
