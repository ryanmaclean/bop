# zam (n├йe rizz)

> Minimal permissive-licensed VM image that reads bop card bundles.
> Chains from EFI. Mounts `.cards/` over 9P. Does 8 things. Ships clean.

---

## Why

- macOS VMs: APSL (not permissive), 30GB+, Apple caps at 2 concurrent on Apple Silicon
- APFS in a non-Apple VM: only permissive implementation is... none. libfsapfs = LGPL. linux-apfs-oot = GPL.
- ZFS: millions of lines, enterprise storage server, CDDL, massive
- Linux kernel: GPL (fine for running bop, wrong for a shippable permissive image)

**Conclusion:** don't read APFS in the VM. The host already does that.
Expose `.cards/` over 9P. The VM never touches APFS.

---

## What it is

A ~3MB VM image (or EFI chainload target) that:

1. Boots
2. Mounts `.cards/` from host via 9P / virtio-fs
3. Reads card bundles (directories + JSON files)
4. Outputs state
5. Exits

No shell. No package manager. No init system. No local disk driver.

---

## The 8 operations

Everything a card reader needs, nothing else:

| Op | Syscall | Notes |
|---|---|---|
| list state dirs | `opendir` / `readdir` | which `.jobcard` dirs exist in `pending/` etc |
| read meta | `open` / `read` / `close` | parse `meta.json` |
| check existence | `stat` | is this card in `running/`? |
| state transition | `rename` | atomic, POSIX, works over 9P |
| create dir | `mkdir` | new card bundle |
| delete | `unlink` / `rmdir` | cleanup |
| scratch space | `tmpfs` | built into kernel, no driver |
| JSON parse | library | jsmn (200 lines, MIT) or serde_json |

Process management (`fork`, `exec`, `kill`) only needed if drip spawns adapters directly.
For read-only inspection: 6 syscalls, zero process ops.

## Kernel choice

| | Unikraft (BSD) | NetBSD (BSD) |
|---|---|---|
| **Use when** | absolute minimum size | reliability, production |
| **Image size** | ~1-3MB | ~20MB+ |
| **UART** | yes | yes |
| **9P client** | `lib/9pfs` | puffs/vnd |
| **TPM driver** | needs porting | tpm(4) in tree |
| **Maturity** | research-grade | 30 years |

Pick Unikraft for the dev/test loop. Graduate to NetBSD when you need it to not fall over.

## Output channel тАФ serial (UART)

The reader binary cannot assume a display, a network stack, or a filesystem to write back to.
**Write everything to serial.** Both Unikraft and NetBSD support UART out of the box.
QEMU captures it with `-serial stdio` тАФ the test harness reads it as plain stdout.

```
zam binary
  тФФтФАтФА reads .cards/ over 9P
  тФФтФАтФА reads PCR[12тАУ15] from TPM
  тФФтФАтФА writes JSON to /dev/ttyS0 (UART)
        тФФтФАтФА QEMU -serial stdio
              тФФтФАтФА test harness reads stdout
```

This is the only output path that works across EFI direct, Unikraft, and NetBSD
without any additional drivers or daemons.

## PCR[12тАУ15] тАФ why they matter

TPM Platform Configuration Registers 12тАУ15 are the "late launch" / application-use PCRs.
In a measured boot chain, they record what software was loaded after the bootloader handed off.
Reading them back to the test harness proves the rizz image booted unmodified and the
card state it reports came from a known-good binary.

```rust
// sketch тАФ read PCR via TPM2 TSS or direct MMIO
let pcr12 = tpm.pcr_read(12)?;
let pcr13 = tpm.pcr_read(13)?;
// ...
let out = json!({ "pcr": [pcr12, pcr13, pcr14, pcr15], "cards": [...] });
uart.write_all(out.to_string().as_bytes())?;
```

TPM access on Unikraft: needs `lib/uk_tpm` or direct MMIO to the vTPM QEMU exposes.
TPM access on NetBSD: `tpm(4)` device, `/dev/tpm0`, standard TSS2 library.

## Two implementation tiers

### Tier A тАФ EFI direct (read-only, no kernel)

[Cosmopolitan libc](https://github.com/jart/cosmopolitan) (MIT) produces a single binary that IS an EFI application. On a Mac, the APFS EFI driver is already in firmware тАФ so a Cosmopolitan binary can `open()` / `readdir()` APFS bundles before any OS loads.

- Zero kernel
- Zero VM
- ~200KB binary
- Read-only (no `rename`, no process management)
- No TPM access (EFI has `EFI_TCG2_PROTOCOL` but it's fiddly)

Use case: boot-time card state inspection, diagnostics, pre-boot task display.

### Tier B тАФ VM (full read/write, state transitions + PCR attestation)

[Unikraft](https://unikraft.org) (BSD) unikernel, configured with:
- `lib/vfscore` тАФ VFS layer
- `lib/9pfs` тАФ 9P client
- `lib/ukcons` тАФ UART console (serial output)
- `lib/posix-process` тАФ only if spawning adapters
- `lib/nolibc` or musl тАФ MIT/BSD

Compile `zam` as the single Unikraft application. Result: ~3-5MB EFI binary.
Boots in ~50ms. Two of these fit in Apple's 2-VM limit.

### Tier C тАФ NetBSD VM (when Unikraft isn't enough)

Full NetBSD with just: `tpm(4)`, 9P client, UART. Serial output same path.
Heavier but `tpm(4)` is battle-tested and the TSS2 stack is mature.

## 9P bridge (host side)

The host (macOS) exposes `.cards/` via:
- `virtio-fs` (if using a hypervisor like QEMU/Apple Virtualization.framework)
- Plan 9 `9pfsd` тАФ trivially small, MIT implementations exist
- Or: `bop` itself speaks 9P as a server, no separate daemon

The VM mounts it at boot as its root filesystem. No local disk required.

---

## Constraints

- 2 VM limit: zam images are stateless and ephemeral. Spin up, read, exit.
- Permissive only: if a dependency is GPL, it doesn't ship in rizz.
- Size: target <5MB image including kernel.
- No APFS knowledge in the VM, ever.
